# 60틱 체결가 기반 MA20 매수 게이트 구현

## 스펙 (사용자 확정)
- **틱 소스**: 거래소 WS `trade` 스트림 실시간 체결가
- **버퍼**: 체결가 60개 (deque maxlen=60, "60틱 기준")
- **MA**: 최근 체결 20개 단순이동평균 (MA20 = "20평균선")
- **매수 게이트**: 현재 체결가 < MA20 → 매수 허용 / 현재 체결가 ≥ MA20 → 매수 금지(대기)
- **버퍼 부족(<20)**: 매수 대기
- **적용 범위**: 업비트/빗썸 모두 (`UpbitWebSocket.WS_URL` 클래스 변수 재사용)
- **동작**: 진입 게이트 only (진행 중인 매수는 유지)

## 구현 단계

### 1. 새 클래스 `TradeTickStream` 추가 (라인 ~420, `UpbitWebSocket` 직후)
- `UpbitWebSocket`과 동일한 패턴 (백그라운드 스레드, 지수 백오프, `_on_open`/`_on_message`/`_on_error`/`_on_close`)
- `WS_URL`은 `UpbitWebSocket.WS_URL`을 참조 (main에서 거래소별 자동 갱신됨)
- 인스턴스 변수:
  - `tick_buffers: {symbol: deque(maxlen=60)}`
  - `current_symbol`, `is_connected`, `_should_reconnect`
- 메서드:
  - `subscribe(symbol)`: ticker와 동일 패턴 — 같은 심볼이면 no-op, 변경 시 재연결
  - `_on_open`: `[{"ticket": ...}, {"type": "trade", "codes": [f"KRW-{symbol}"]}]` 전송
  - `_on_message`: `code`에서 심볼 추출 → `trade_price`를 `tick_buffers[symbol]`에 append
  - `get_ma20(symbol)`: 버퍼 길이 < 20이면 None, 아니면 마지막 20개 평균 반환
  - `get_last_price(symbol)`: 버퍼가 비었으면 None, 아니면 마지막 값 반환

### 2. 전역 인스턴스 생성 (라인 ~877 `RealMarketData._ws` 근처)
```python
trade_ws = TradeTickStream() if WEBSOCKET_AVAILABLE else None
```

### 3. 헬퍼 함수 `check_tick_ma_gate(symbol)` 추가 (`RealMarketData` 클래스 내 정적 메서드 또는 모듈 함수)
- `trade_ws` 미가용 → 매수 대기 (False 반환 + 경고 로그)
- `trade_ws.get_ma20(symbol)`이 None (버퍼 < 20) → 매수 대기 (False 반환)
- `last_price = trade_ws.get_last_price(symbol)` vs `ma20`:
  - `last_price < ma20` → 통과 (True)
  - `last_price >= ma20` → 대기 (False)

### 4. 메인 루프 게이트 삽입 (라인 3275-3278 사이)
- 기존 `RealMarketData.subscribe_websocket(symbol)` 호출 후 `trade_ws.subscribe(symbol)` 추가
- `should_place_buy_orders()` 진입 블록 맨 앞(`cancel_buy_orders()` 호출 전)에 게이트 체크:
  ```python
  if not check_tick_ma_gate(symbol):
      print_log(LogLevel.INFO, f"매수 대기 — 60틱 MA20 게이트 미충족 ({symbol})")
      time.sleep(1)
      continue   # 다음 사이클에서 재평가
  ```
- 로그에 MA20 값과 현재 체결가를 함께 출력하여 디버깅 용이하게 함

## 동작 흐름 (변경 후)
```
사이클 시작
  ├─ 심볼 확정
  ├─ ticker WS 구독 (기존)
  ├─ trade WS 구독 (신규)
  ├─ should_place_buy_orders() → True
  │    ├─ check_tick_ma_gate() → False: 로그 + sleep(1) + continue (매수 대기)
  │    └─ check_tick_ma_gate() → True: 기존 매수 로직 진행
  └─ 매수/매도/스탑로스 (기존 로직 유지)
```

## 영향 받지 않는 부분
- 진행 중인 매수/매도 로직 (DynamicBuyOrder, SellController) — 그대로 유지
- `MarketAnalyzer`의 캔들 기반 ma60/is_below_ma60 — 독립적으로 유지 (틱 MA와 별개)
- VolatilityScanner, command.txt 감시 — 영향 없음

## 비고
- WS 연결이 끊어지면 버퍼가 갱신되지 않으므로, 버퍼가 20개 이상이라도 마지막 갱신으로부터 일정 시간이 지나면 신뢰성이 떨어짐. 필요시 마지막 tick 타임스탬프를 추적해 stale 체크 추가 가능(초기 구현에서는 제외, 단순화).
- `trade` 스트림은 ticker보다 메시지 빈도가 높으므로 CPU 약간 증가 가능하지만, deque append는 O(1)이라 핫 루프에 미치는 영향 미미.
