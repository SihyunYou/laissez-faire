## 대규모 지연 최적화 계획

분석 결과 핫 루프 1회전당 발생하는 API 호출:
- **/v1/accounts: 5~6회** (AccountChecker 남용) → Private WS `myAsset`으로 0회
- **GET /v1/order: pending/tracking 수만큼** (per-order 조회) → Private WS `myOrder`/`myTrade`로 0회
- **GET /v1/candles: 4~6회** (MarketAnalyzer 중복 생성) → 캐싱으로 1~2회
- **디스크 IO: command.txt 매 사이클 2회 + state.txt 동기 쓰기**

---

### A. Private WebSocket 도입 — `UpbitPrivateWS` 클래스 신규

`wss://api.upbit.com/websocket/v1/private` 연결. JWT(`HS512`, `access_key`+`nonce`)를 `Authorization: Bearer` 헤더로 전송. 한 연결에서 `myAsset` + `myOrder` 동시 구독.

**구독 데이터 처리:**
- **myAsset** 메시지: `{currency, balance, locked, avg_buy_price}` → 잔고 캐시(dict by currency) 실시간 갱신
- **myOrder** 메시지: `{uuid, state, executed_volume, executed_funds, ...}` → 주문 상태/체결 이벤트 큐 + 상태 캐시(dict by uuid) 갱신
- **myTrade** 메시지: 체결 이벤트 → 콜백 트리거

**API**:
- `get_symbol_info(symbol)` → 캐시에서 즉시 반환 (balance, locked, avg_buy_price)
- `get_krw_balance()` → 캐시에서 KRW 잔고 즉시 반환
- `get_owned_symbols()` → 캐시 keys
- `get_order_state(uuid)` → 캐시에서 즉시 반환
- `wait_order_event(uuid, timeout)` → 이벤트 큐에서 해당 UUID 체결 대기
- 초기 부팅 시 REST /v1/accounts 1회로 캐시 seed (이후 WS push로만 갱신)

**재연결**: 백그라운드 데몬 스레드, 지수 백오프. 연결 끊김 시 캐시 무효화 + REST 폴백 시드 재수행.

---

### B. AccountChecker를 WS 캐시 래퍼로 전환

`AccountChecker.__init__`에서 매번 /v1/accounts 호출하던 것을, 전역 `UpbitPrivateWS` 인스턴스의 캐시에서 읽도록 변경.

**변경점** (7곳):
- `SellController`: `has_holdings`(1729), `get_avg_buy_price`(1739), `get_total_volume`(1744), `get_available_volume`(1749) — 모두 WS 캐시 조회로 O(1)
- 메인 루프: `balance, locked, avg = AccountChecker().get_symbol_info(symbol)`(2747) — WS 캐시
- 부팅(2537), 사이클 종료(2813) — 초기/종료 시만 REST 1회씩 (정확한 잔고 필요)

**안전장치**: WS 미연결 시 기존 REST 폴백 유지. 캐시 TTL(예: 5초) 초과 시 REST로 재동기화 옵션.

### C. 주문 체결 추적 — per-order GET /v1/order 제거

**`DynamicBuyOrder` 측:**
- `_check_order_execution`(1219), `_check_and_apply_plan_shift` 내 `_get_order_info`(1106) 호출 → WS 주문 상태 캐시 조회로 전환
- `_get_order_info`(1402): WS 캐시 우선, 미스 시 REST 폴백 유지 (레거시 호환)

**`SellController.check_sell_fills`(1913):**
- 현재 tracking의 모든 entry마다 GET /v1/order 호출 → WS `myOrder`/`myTrade` 이벤트 큐 기반으로 전환
- 체결 이벤트 수신 시 entry.filled=True 처리 (API 호출 0회)

**배치 처리 도입 (WS 미수신 시 폴백):**
- 현재 `GET /v1/order?uuid=X`를 N회 → `GET /v1/orders?uuids=X,Y,Z` 1회 일괄 조회로 대체
- `_get_orders_batch(uuids)` 신규 헬퍼 (업비트 복수 UUID 조회, 초당 30회)

### D. 캔들 데이터 캐싱/통합

**`CandleCache` 클래스 신규** (또는 모듈 전역):
- symbol별 최근 200캔들 데이터 + 타임스탬프 캐싱
- TTL 기반 만료 (예: 캔들 주기 1분이므로 30초 TTL)
- 한 사이클에 `MarketAnalyzer(symbol)` 4회 생성 → 캐시 히트로 REST 0회 (TTL 내)
- `VolatilityProtector.check_volatility_protection`도 동일 캐시 재사인 (60캔들 = 캐시에서 슬라이스)

**변경점:**
- `CandleInfoFetcher.__init__`(2108): 캐시 확인 → 히트 시 API 호출 스킵
- `MarketAnalyzer` 생성 4회(2589,2640,2658,2660) → 캐시로 1회만 REST
- `VolatilityProtector`(2144, 1608/2582/2611/2417): 동일 캔들 캐시 재사용

### E. 디스크 IO 제거/통합

1. **command.txt 중복 읽기 제거**:
   - `check_command_file()`(1597)에서 읽고 캐싱 → 메인 루프(2597)는 캐시된 값 재사용
   - 또는 메인 루프의 별도 파일 읽기를 `check_command_file()` 통합

2. **state.txt / balance.txt 비동기화**:
   - 현재 동기 `open(...,'w')` → 별도 데몬 스레드 큐 기반 비동기 쓰기로 전환
   - 또는 빈도가 낮으므로 그대로 두되 핫 루프 내 호출 제거

3. **`log_state` / `log_balance` 호출 빈도 점검**: 핫 루프 내 중복 호출 제거

---

### F. rate_limiter / safe_api_call 정비

- `rate_limiter` interval=0(무효) 상태 유지하되, 429 응답 처리를 `safe_api_call`로 올림:
  - HTTP 429 수신 시 백오프 후 재시도 (현재는 get_current_price만 처리)
- JWT 토큰 생성 함수 공통화 (현재 각 메서드마다 중복 생성)

### 우선순위 및 순서

1. **[P0] A + B**: Private WS + AccountChecker 전환 — 최대 병목(5~6회/iteration → 0회)
2. **[P0] C**: 주문 체결 WS 이벤트 + 배치 조회 — 두 번째 병목
3. **[1] D**: 캔들 캐싱 — 세 번째 병목
4. **[2] E**: 디스크 IO 정리
5. **[2] F**: rate_limiter/429 정비

### 예상 효과
- 핫 루프 1회전 API 호출: **약 15~20회 → 0~2회** (WS push 기반)
- 루프 지연: 수백ms~수초 → **밀리초 단위** (캐시/WS 조회는 O(1))
- 디스크 IO: 매 사이클 4~6회 → **0~1회**
